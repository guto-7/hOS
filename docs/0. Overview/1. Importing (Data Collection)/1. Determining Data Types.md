### Determining Data Types

Before a node can process anything, it must define exactly what data it will accept. This means identifying the diagnostic sub-mediums relevant to the domain, the file formats they produce, and why each is included.

---

### Diagnostic Sub-Mediums

Some domains have a single data source (e.g., blood work produces lab reports). Others encompass multiple sub-mediums that each produce fundamentally different data (e.g., radiology includes X-ray, MRI, CT, ultrasound — each with different output formats, resolutions, and diagnostic applications).

##### Each node must:
- **Identify all sub-mediums** within its diagnostic domain.
- **Research which are most commonly used** in clinical practice and why. Prioritise sub-mediums that are widely available, routinely ordered, and produce data with established evaluation criteria.
- **Determine which sub-mediums the node will support** and document the rationale. Not every sub-medium needs to be included — but exclusions must be deliberate and documented in [Limitations](2.%20Limitations.md).

---

### File Formats

##### For each included sub-medium, define:
- **What file format(s) the data arrives in** (PDF, DICOM, CSV, manual entry, etc.).
- **Whether a single format or multiple formats** need to be supported. If multiple, document why each is necessary and how they differ.
- **Which format is the primary target** for initial implementation. A node may eventually support several formats, but must start with the most common and most reliable.

---

### Research Requirements

Data type decisions must be evidence-based. Each node should document:

- **Prevalence** — how commonly is this data type produced in standard clinical workflows?
- **Standardisation** — is there an established format or does it vary by provider/lab/manufacturer?
- **Downstream viability** — can this data type be reliably parsed (Layer 2) and evaluated (Layer 3) with deterministic methods?

A data type that is common but unstandardised may require more work at the Formatting and Unifying stages. A data type that cannot be deterministically evaluated may not belong in the node at all.

---

## Node Implementation

When applying this framework to a specific node, document the following in that node's Importing section:

###### Sub-Mediums
- List each sub-medium the node supports.
- For each, provide: what it is, why it's included, how common it is in clinical practice, and what established evaluation criteria exist for it.
- If the domain only has one sub-medium, state this explicitly.

###### File Formats
- For each sub-medium, list the accepted file format(s).
- Identify the primary format for initial implementation.
- Note any provider/lab/manufacturer variation in how the format is produced.

###### Sources
- Reference the clinical literature, standards bodies, or established guidelines that informed these decisions.
