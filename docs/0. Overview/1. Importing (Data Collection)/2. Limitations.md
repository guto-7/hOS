### Limitations

Every node must explicitly define what it will not accept. Limitations are not afterthoughts — they are deliberate boundaries that protect the reliability of everything downstream. A node that tries to handle data it cannot deterministically evaluate will produce unreliable diagnostics.

---

### Why Define Limitations Early

The Evaluation layer (Layer 3) is deterministic. It can only analyse data that has known clinical criteria and established reference standards. If data enters the pipeline that cannot be evaluated against clear, evidence-based benchmarks, it undermines the core principle of the system — that diagnostics are not probabilistic.

Limitations defined here flow forward through every layer. If a data type is excluded at Importing, it never reaches Parsing, Evaluation, or Insight. This is by design.

---

### What to Exclude

##### Each node must identify and document exclusions across three categories:

**Data Types**
- Sub-mediums or file formats that exist within the domain but are not supported. This may be due to low prevalence, poor standardisation, or lack of deterministic evaluation criteria.

**Scope**
- Diagnostic questions that fall outside the node's domain, even if the data could technically contain relevant information. A blood work node should not attempt to interpret imaging artifacts found in a lab report PDF.
- Edge cases where data technically fits the accepted format but is too ambiguous, incomplete, or unreliable to process with confidence.

**Quality**
- Minimum data quality thresholds below which the node rejects input rather than attempting to process it. Corrupted files, illegible scans, reports missing critical fields.
- The node should fail clearly rather than produce uncertain output.

---

### Deferred Data Types

Experimental or emerging data types that do not yet have established clinical benchmarks but may become viable as standards mature. These are not excluded permanently — they are parked. Each deferred type should be documented with the reason it is not yet supported and the conditions under which it would be reconsidered (e.g., clinical benchmarks are published, standardised formats emerge, evaluation methods become deterministic).

---

## Node Implementation

When applying this framework to a specific node, document the following in that node's Importing section:

###### Excluded Data Types
- List each excluded sub-medium or format with a clear rationale for why it is not supported.

###### Deferred Data Types
- List emerging or experimental data types that are not currently supported, with the criteria that would trigger reconsideration.

###### Scope Boundaries
- Define what diagnostic questions the node does and does not answer, even when the data might overlap with other domains.

###### Quality Thresholds
- Define the minimum requirements for accepted input — what constitutes a valid submission and what triggers rejection.
