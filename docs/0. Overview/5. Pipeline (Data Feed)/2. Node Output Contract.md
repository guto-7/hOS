### Node Output Contract

The standardised structure that all node output must conform to. The output contract is the single schema that both the frontend and the Orchestrator consume — different consumers may use different slices of it, but the underlying structure is the same. Without a consistent contract, adding new nodes or extending existing ones breaks downstream consumers.

---

### What the Contract Must Include

##### Identity

- Which node produced this output. Node identifier, version, and timestamp of when the output was generated.
- Which version of the node's evaluation criteria was used. As condition registries, domain scores, and analysis engines evolve, the contract must track what produced the findings.

##### Unified Data

- The standardised data from Layer 2 — canonical identifiers, standardised values, units, and initial flags. This is the raw evaluated dataset that both the frontend and the Orchestrator may need to reference directly.

##### Evaluation Outputs

- **General Analysis** — critical flags surfaced, domain groupings with their summary status.
- **Domain Scores** — each composite metric with its inputs, score, and scoring system version.
- **Condition Mappings** — each condition evaluated, which criteria were met/not met/not assessed, and the diagnostic method used.
- **Diagnostic Certainty** — certainty grades for each finding, incompleteness disclosures, and weighting data.

##### Insight Outputs

- Contextualised explanations produced by the LLM layer, scoped to the findings they reference. Stored separately from evaluation outputs — clearly marked as LLM-generated, never mixed with deterministic data.

##### Metadata

- Data completeness summary — what proportion of expected data was available, what was resolved by the user, what remains missing.
- Processing log — which pipeline stages ran, any errors or warnings, any user resolutions applied during validation or certainty resolution.

---

### Design Principles

**Versioned**
- The contract schema itself must be versioned. As the Evaluation layer grows, the contract grows — but older outputs must remain readable. Breaking changes require a version increment.

**Self-describing**
- Each output must carry enough metadata that a consumer can understand what it contains without prior knowledge of the specific node. The Orchestrator should be able to process a new node's output from the contract structure alone.

**Comprehensive**
- The contract is almost all-encompassing by design. It includes nearly everything the node produces — unified data, all evaluation outputs, insight artifacts, and full metadata. The Orchestrator does not read from node storage directly, so the contract must contain everything it could need for cross-diagnostic analysis. Consumers query what they need from the contract; not everything is used every time, but everything is available.

**Deterministic and LLM data separated**
- Evaluation outputs and Insight outputs must be structurally distinct within the contract. A consumer must be able to access deterministic findings without encountering LLM-generated content, and vice versa.

---

### Node Implementation

When applying this framework to a specific node, document the following in that node's Pipeline section:

###### Contract Schema
- Define the complete output structure for this node — all fields, types, and nesting. Map each field to the pipeline layer that produces it.

###### Versioning Strategy
- Define how contract versions are tracked and how schema changes are handled.

###### Node-Specific Fields
- Document any fields unique to this node's domain that extend the base contract. These must be structured consistently with the base schema so the Orchestrator can process them without node-specific logic.
