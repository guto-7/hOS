### Orchestrator Data Feed

The Orchestrator Data Feed defines what each node sends up to the Medical Orchestrator and how it is packaged for cross-diagnostic consumption. This is the input to the Orchestrator's own Import layer — the Orchestrator treats every node's feed exactly as a diagnostic node treats a raw data source. The feed must be comprehensive enough to power the big data engine without the Orchestrator ever accessing node storage directly.

---

### What the Orchestrator Receives

The Orchestrator receives the full output contract from each node. This is almost all-encompassing by design — the Orchestrator cannot go back and request additional data from a node's storage, so the contract must contain everything it could need.

##### Unified Data
- The complete standardised dataset from the node — canonical identifiers, values, units, initial flags, and severity quantification. The Orchestrator needs raw values to run its own cross-diagnostic evaluations, not just the node's conclusions.

##### Evaluation Outputs
- General analysis groupings and critical flag data.
- Domain scores with inputs and scoring system versions.
- Condition mappings with full criteria detail — met, not met, not assessed.
- Diagnostic certainty grades and weighting data for every finding.

##### Incompleteness Data
- What data was missing, what the user resolved, what remains incomplete. The Orchestrator must know the evidential strength of what it is ingesting — it cannot treat a low-certainty blood work finding the same as a high-certainty one when building systemic evaluations.

##### Node Identity and Versioning
- Which node produced the output, which version of the evaluation criteria was used, and when the output was generated. The Orchestrator must track provenance across all nodes to maintain auditability of its own cross-diagnostic findings.

---

### How the Feed Is Structured

##### Consistent Across Nodes
- Every node's feed must conform to the same structural pattern defined in the [Node Output Contract](2.%20Node%20Output%20Contract.md). The Orchestrator should be able to ingest a new node's output from the contract structure alone — no node-specific parsing logic.
- Node-specific fields (unique data types, domain-specific scores) must extend the base contract consistently, not break it.

##### Self-Describing
- Each feed must carry enough metadata that the Orchestrator can understand what it contains, what domain it represents, and how to weight it against feeds from other nodes. A new node added to the system should be consumable by the Orchestrator without Orchestrator-side code changes.

##### Versioned
- As nodes evolve their evaluation criteria and output contracts, the Orchestrator must handle multiple contract versions. The feed includes the contract version so the Orchestrator can process it correctly and flag when a node's output contract has changed in ways that affect cross-diagnostic evaluations.

---

### How the Orchestrator Uses the Feed

The Orchestrator ingests node feeds as its own Import layer and runs the same 5-layer pipeline:

- **Import** — receives output contracts from all nodes.
- **Unify** — normalises cross-domain data into a unified systemic view.
- **Evaluate** — runs cross-diagnostic analysis. Conditions that span multiple domains, systemic interactions, patterns that no single node can see.
- **Insight** — contextualises systemic findings using the Orchestrator's own LLM skill.
- **Pipeline** — stores cross-diagnostic evaluations, feeds the Orchestrator's own dashboard and chat.

The comprehensiveness of the node feeds is what makes this possible. The Orchestrator's evaluation depth is directly limited by what the nodes send it.

---

### Node Implementation

When applying this framework to a specific node, document the following in that node's Pipeline section:

###### Feed Contents
- Define the complete data payload sent to the Orchestrator. Map each element to the output contract field it corresponds to.

###### Node-Specific Extensions
- Document any domain-specific fields that extend the base contract and how they are structured for Orchestrator consumption.

###### Feed Trigger
- Define when the feed is sent — on every new evaluation, on data change, on user request, or on a defined schedule. Document the rationale.

###### Versioning
- Define how contract version changes are communicated to the Orchestrator and how backwards compatibility is maintained.
