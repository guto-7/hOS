### Data Storage

All data produced by a node is stored within that node's own compartment. Nodes do not share storage. This is the foundation of the Pipeline layer — before data can be packaged into output contracts or routed to consumers, it must be persisted in a structured, isolated, and scalable way.

---

### Compartmentalised Storage

Each node owns its storage entirely. A blood work node's data lives in its own space, separate from imaging, separate from body composition. The Orchestrator maintains its own compartment for cross-diagnostic evaluations.

#### Why Compartmentalised

- **Scalability** — each node's storage grows independently. Adding a new node means adding a new compartment, not restructuring a shared store.
- **Isolation** — a schema change in one node's storage does not affect any other node. Migrations are node-scoped.
- **Clarity** — there is no ambiguity about where data lives or which node owns it. Every data point has one home.
- **Modularity** — if a node is removed or replaced, its storage goes with it. No orphaned data in shared tables.

---

### What Is Stored Per Node

##### Unified Data
- The standardised, validated, and flagged data from Layer 2. Canonical identifiers, standardised values and units, initial flags, severity quantification, and provenance metadata. This is the node's internal source of truth for all raw diagnostic data.

##### Evaluation Outputs
- General analysis groupings and critical flag data.
- Domain scores with inputs, formulas, and scoring system versions.
- Condition mappings with criteria met/not met/not assessed and diagnostic method references.
- Diagnostic certainty grades, weighting data, and incompleteness disclosures.

##### Insight Artifacts
- LLM-generated contextualisation and chat history. Stored separately from deterministic data — structurally distinct so that evaluation outputs are never mixed with probabilistic content.

##### Processing Metadata
- Upload records, validation resolutions, user corrections applied during the pipeline, and processing logs. The complete audit trail of how data moved through the pipeline.

##### Raw Input Archive
- Every original input must be stored in its own system within the node's compartment — the original uploaded file (PDF, DICOM, etc.), exactly as it was received. This is never modified or overwritten.
- Every user action that altered data during the pipeline must be traceable — validation corrections, manual data entry, certainty resolution decisions, resubmissions. Each action is logged with what was changed, from what, to what, and when.
- This creates a complete chain from original source to final evaluation output. Any finding can be traced back through every transformation and user interaction to the raw input that produced it.

---

### Storage Boundaries

##### Nodes Do Not Read Each Other's Storage
- No node has access to another node's compartment. Cross-domain data access happens exclusively through the [Orchestrator Data Feed](4.%20Orchestrator%20Data%20Feed.md) — packaged, versioned output contracts, never raw storage.

##### The Orchestrator Does Not Read Node Storage
- The Orchestrator consumes what nodes send it through the output contract. This is a deliberate boundary — the Orchestrator works with packaged, versioned output, not node internals. If the Orchestrator needs data, it must be in the contract.

##### Storage Mechanism Is Node-Specific
- Each node defines its own storage approach based on its data type. Tabular data may use a relational store. Binary data (images, scans) may use file-based storage with metadata references. The mechanism is an implementation decision made per node — the compartmentalisation principle applies regardless.

---

### Node Implementation

When applying this framework to a specific node, document the following in that node's Pipeline section:

###### Storage Architecture
- Define the storage mechanism(s) used — relational database, file storage, or a combination. Document why the chosen approach fits the node's data type.

###### Compartment Schema
- Define the complete storage structure — tables, fields, relationships, file organisation. Map each stored element to the pipeline layer that produces it.

###### Separation of Deterministic and LLM Data
- Define how evaluation outputs and insight artifacts are structurally separated in storage.

###### Audit Trail
- Define what processing metadata is stored and how it supports traceability from raw import to final evaluation output.

###### Raw Input Archive
- Define how original uploads are stored, how user actions are logged, and how the traceability chain from source to output is maintained.
