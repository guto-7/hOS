### History & Longitudinal Tracking

History is not a separate system. It is a property of data that already flows through the pipeline. Every piece of data has a point in time — by making that timestamp a first-class field from import through to output, longitudinal tracking becomes a query pattern, not an additional architecture.

---

### Collection Date as a Required Field

Every dataset entering the pipeline must carry a collection date — when the diagnostic data was originally collected, not when it was uploaded to the system.

##### At Import

- The collection date is extracted from the raw input during Importing, alongside other metadata. For blood work, this is the test date printed on the lab report. For imaging, the acquisition date in the DICOM metadata. For body composition, the measurement date.
- If the raw input does not contain a collection date, the user is prompted to provide one. This is surfaced through the same resolution interface used for validation failures — the pipeline does not proceed without a date.
- A default option of the input date (date of upload) must be available for cases where the user cannot determine the original collection date. This default must be clearly marked as an input date, not a confirmed collection date, so it can be corrected later.

##### At Storage

- The collection date is assigned to the entire dataset from that input — all data points from the same upload share the same collection date. This groups them as a single snapshot in time.
- The upload date is stored separately. The distinction between collection date and upload date is maintained from day one — a user may upload a report months after it was produced, and the system must know the difference.

##### At Output

- The collection date is a core field in the [Node Output Contract](2.%20Node%20Output%20Contract.md). Every output carries the date of the data it was evaluated from.
- This means any consumer — the node's own frontend, the Orchestrator — can sort, filter, group, and build timelines from the contract alone. No additional queries or lookups required.

---

### What This Enables

##### Trend Detection

- With every dataset timestamped, comparing the same data point across multiple records is a straightforward query. Tracking ferritin over five blood tests becomes: retrieve all ferritin values, order by collection date.
- Trend computation (rising, falling, stable, rate of change) is applied at the Evaluation layer using this timestamped data. The Pipeline layer's responsibility is ensuring the dates are there and correct — not computing the trends themselves.

##### Historical Evaluation Comparison

- Not just raw data trends, but comparison of evaluation outputs over time. How a domain score changed between two uploads. Whether a condition that was flagged six months ago is still flagged or has resolved.
- This requires evaluation outputs to be stored per collection date, not overwritten. Each upload produces its own evaluation — historical evaluations are preserved alongside the data that produced them.

##### Cross-Node Timelines (Orchestrator)

- The Orchestrator receives collection dates from every node's output contract. This allows it to correlate timelines across domains — blood work changes that coincide with imaging changes, body composition shifts that align with metabolic marker trends.
- Timeline correlation is an Orchestrator-level evaluation, but it depends entirely on every node providing accurate, consistent collection dates.

---

### Node Implementation

When applying this framework to a specific node, document the following in that node's Pipeline section:

###### Date Extraction
- Define where and how the collection date is extracted from the node's raw input. Document the expected location in each accepted file format.

###### Missing Date Handling
- Define how the user is prompted when no collection date is found, and how the input date default is applied and marked.

###### Snapshot Grouping
- Define how data from a single input is grouped under one collection date and how multiple uploads on the same collection date are handled.

###### Historical Preservation
- Define how evaluation outputs are stored per collection date so that historical evaluations are never overwritten.
