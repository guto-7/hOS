### Frontend Data Feed

Each node owns its own dashboard and chat interface. The Frontend Data Feed defines what data is sent to these interfaces, how it is structured for rendering, and how the pipeline ensures the frontend always has what it needs to display the node's full diagnostic picture.

---

### What the Frontend Receives

The frontend consumes a slice of the node's output contract, formatted for rendering. This includes:

##### Dashboard Data

- **Critical flags** — individually flagged data points at critical severity, surfaced first and prominently.
- **Grouped data** — all data points organised into domain groupings as defined in General Analysis, with per-group summary status.
- **Domain scores** — composite metrics with their values, scales, and contextual information for display.
- **Condition findings** — conditions with criteria met/not met/not assessed, certainty grades, and incompleteness disclosures.
- **Validation resolutions** — pending items requiring user action (failed validation, incomplete data for certainty scoring).

##### Chat Data

- **Evaluation context** — the Insight layer's contextualised explanations, scoped to specific findings.
- **Conversation history** — prior chat threads within this node, managed per-node.
- **Available data scope** — what data and findings the chat can reference, bounded to this node's domain.

---

### How Data Is Structured for Rendering

##### Formatting

- Pipeline data is stored in a scalable, processable format (defined in [Data Storage](1.%20Data%20Storage.md)). The frontend feed transforms this into rendering-ready structures — display values, labels, colour mappings, sort orders, and grouping hierarchies.
- Formatting logic lives in the Pipeline layer, not in the frontend. The frontend receives data that is ready to render — it does not perform its own data transformation or evaluation logic.

##### State Management

- The frontend must reflect the current state of the pipeline. If a user submits a validation correction or provides missing data for certainty resolution, the feed must update the affected findings and their display state.
- The feed must distinguish between complete and in-progress evaluations — data that is still pending user resolution should be clearly marked in the frontend state.

##### Incremental Updates

- The feed should support updating individual sections without requiring a full re-render of all data. A resolved validation issue should update the affected data points and any downstream findings, not reload the entire dashboard.

---

### Node Implementation

When applying this framework to a specific node, document the following in that node's Pipeline section:

###### Dashboard Structure
- Define what data is sent to the dashboard, how it is grouped and ordered, and the rendering format for each evaluation type.

###### Chat Context
- Define what evaluation data and findings are available to the chat interface, and how conversation scope is bounded to the node's domain.

###### Formatting Rules
- Define how stored data is transformed into rendering-ready structures — display values, labels, colour mappings, and grouping hierarchies.

###### State Updates
- Define how the frontend feed handles pipeline state changes — validation resolutions, certainty updates, and incremental re-evaluation.
