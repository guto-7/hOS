### Data Routing

Before any evaluation runs, unified data must be routed to the correct analysis engines in the format each one requires. The Unifying layer standardises data for storage and consistency — but different evaluation methods may require different input shapes. A rule engine expects tabular key-value pairs. An image classifier expects pixel data in a specific tensor format. A scoring algorithm may need normalised values on a defined scale.

Data Routing sits between unified storage and evaluation execution. It does not alter the data — it transforms the *representation* for each downstream engine.

---

### What Data Routing Covers

##### Engine Registry

- Which evaluation engines does this node use? Each engine defined across General Analysis, Domain Analysis, Condition Mapping, and Diagnostic Certainty is a consumer of routed data.
- The routing layer must know what engines exist and what each one expects as input.

##### Format Transformation

- For each engine, define how unified data is transformed into the required input format. This may be trivial (passing structured records directly to a rule engine) or significant (converting stored image data into tensors with specific dimensions, colour spaces, and normalisation).
- Transformations must be deterministic and lossless where possible — the same unified data must produce the same engine input every time.

##### Routing Logic

- Which data goes to which engine. Not all data goes to every engine — a condition mapping engine for hormonal conditions only needs hormonal markers, not the full dataset.
- Routing must be efficient. Engines should receive only the data they need, in the format they need, without redundant processing.

---

### Why This Is Separate From Unifying

Unifying produces one canonical representation for storage and consistency. Routing produces multiple representations for consumption by different engines. These are different concerns:

- **Unifying** answers: what does this data look like in our system?
- **Routing** answers: what does this data need to look like for each engine to process it?

Keeping them separate means the storage format can evolve independently of evaluation input requirements, and new engines can be added without changing how data is stored.

---

### Node Implementation

When applying this framework to a specific node, document the following in that node's Evaluation section:

###### Engine Registry
- List all evaluation engines the node uses and the input format each one requires.

###### Format Transformations
- For each engine, define the transformation from unified data to engine input. Document any data shaping, normalisation, or restructuring required.

###### Routing Map
- Define which data subsets are routed to which engines and the logic that determines routing.
