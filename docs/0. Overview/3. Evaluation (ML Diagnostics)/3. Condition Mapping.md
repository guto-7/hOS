### Condition Mapping

The deepest analytical layer. Where General Analysis surfaces individual flags and Domain Evaluations scores broad health areas, Condition Mapping identifies specific clinical conditions by detecting defined patterns across multiple data points. This is multi-variable pattern recognition — the system matching combinations of values against established diagnostic criteria.

---

### What Condition Mapping Is

A condition is not a single abnormal value. Iron deficiency is not just low ferritin — it is a pattern across ferritin, serum iron, TIBC, and transferrin saturation. Each condition the node can identify must be defined as a set of criteria — which data points are involved, what values or flag states satisfy the criteria, and how many must be met.

**Condition mapping is scoped to the node's own data.** A blood work node evaluates conditions using blood markers only. It does not pull in waist circumference from body composition or imaging findings from radiology — even when a clinical condition's full diagnostic criteria span multiple domains (e.g., metabolic syndrome requires both blood markers and anthropometric measurements). Cross-domain condition evaluation is the Orchestrator's responsibility.

**Big Data Engine**. The Orchestrator function  — it ingests unified data and diagnostic outputs from every node and performs its own evaluation layer. Conditions that require data from multiple nodes, systemic interactions, and cross-diagnostic pattern recognition all live there. This is by design the most complex evaluation in the system, and it scales with every node added.

---

### What Condition Mapping Is Not

- **Not a diagnosis — yet.** The system identifies that a defined set of criteria are met. It does not tell the user they have a disease. That distinction is critical — clinically, legally, and in how findings are communicated to the user. However, the long-term goal is full regulatory approval. Every system in this layer is built to be provable, auditable, and certifiable — first through TGA approval, then FDA. The deterministic architecture exists specifically to make this path possible.
- **Not exhaustive — yet.** A node maps conditions for which established, evidence-based criteria exist. Conditions without clear diagnostic criteria do not belong here. As the system matures and gains regulatory approval, the condition registry will expand — but only with the same standard of evidence-based, deterministic criteria.
- **Not probabilistic.** Criteria are either met or not met. The strength of the finding given incomplete data is handled by [Diagnostic Certainty](4.%20Diagnostic%20Certainty.md), but the criteria evaluation itself is binary per criterion.

---

### What Each Node Must Define

##### Condition Registry

- The complete list of conditions this node can identify. Each condition must be clinically established with published diagnostic criteria.
- For each condition: what it is, which clinical domain it belongs to (mapping to the groupings defined in [General Analysis](1.%20General%20Analysis.md)), and why it is included.
- The registry must be versioned. As clinical guidelines evolve, criteria change — the system must track which version of the criteria produced a given finding.

##### Criteria Definitions

- For each condition, the exact set of data points involved and the thresholds or states that satisfy each criterion.
- Which criteria are required (must be met for the condition to be flagged) and which are supporting (strengthen the finding but are not mandatory).
- How criteria interact — are they additive (meet X out of Y), sequential (A must be true before B is evaluated), or weighted?

##### Diagnostic Method

- What computational approach is used to evaluate conditions against the data? Rule engines, decision trees, weighted scoring matrices, classical ML classifiers — the method must be documented.
- Why this method is appropriate for the conditions being mapped. Simple threshold-based conditions may only need a rule engine. Conditions with complex interactions between many variables may require a more sophisticated approach.
- The method must be deterministic and reproducible. Document how the implementation is tested and validated against known cases.

##### Cross-Condition Interactions

- Some conditions share data points or have known clinical relationships. The node must document where conditions overlap and how this is handled.
- A single set of abnormal values may satisfy criteria for multiple conditions. This is expected and valid — the system should report all conditions whose criteria are met, not attempt to select one.
- Contradictory conditions (where the same data could indicate mutually exclusive states) must be documented and surfaced clearly.

##### Storage

- Each condition finding must be stored with: the condition identified, which criteria were met and which were not, the data points and values that contributed, the criteria version used, and the diagnostic certainty level.
- Stored findings must be granular enough to explain *why* a condition was flagged — not just that it was.

### Node Implementation

When applying this framework to a specific node, document the following in that node's Evaluation section:

###### Condition Registry
- List all conditions the node identifies. For each: clinical definition, the domain grouping it belongs to, published diagnostic criteria source, and rationale for inclusion.

###### Criteria Definitions
- For each condition, document the exact criteria — data points, thresholds, required vs supporting criteria, and interaction logic.

###### Diagnostic Method
- Document the computational approach used, why it was chosen, and how it is validated against known cases.

###### Cross-Condition Map
- Document known overlaps, shared data points, and mutually exclusive conditions within the registry.

###### Storage Format
- Define how condition findings and their constituent evidence are stored.

###### Sources
- Reference the clinical guidelines, diagnostic criteria publications, and classification systems used.
