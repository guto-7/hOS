### Diagnostic Certainty

Not every evaluation has complete data. A condition may require five data points to fully assess, but only three are present. A domain score may be missing one of its key inputs. The system must still produce useful output — but it must be transparent about how complete the evidence is behind any finding.

---

### Resolving Incomplete Data

Before certainty can be scored, missing data must be addressed.

##### User Resolution

- Incomplete evaluations are flagged to the user through the node's dashboard — similar to the resolution interface in [Validation](../2.%20Unifying%20(Parsing%20&%20Sorting)/2.%20Validation.md). The user is shown what data is missing, which evaluations are affected, and how completeness impacts the findings.
- The user can provide missing data (e.g., from a different report or manual entry), confirm proceeding with partial data, or dismiss the evaluation entirely.
- This ensures the user is aware of gaps before the system makes any adjustments for incompleteness.

##### Data Completeness

- After user resolution, the system recalculates what proportion of required data points are present for each evaluation. Some gaps may have been filled, others confirmed as unavailable.
- Completeness is calculated at every level — per condition, per domain score, and per panel grouping.
- Where data remains incomplete, the system applies weighted adjustment — missing data points are not treated equally. A missing primary indicator degrades the evaluation more than a missing supporting marker. The weighting used here aligns with the same clinical weights defined in Certainty Scoring.

##### Criteria Fulfilment

- Of the data points that are present after resolution, how many met the criteria? The distinction between "not met" and "not assessed" is critical. A criterion that was evaluated and failed is evidence against. A criterion that could not be evaluated due to missing data is unknown — not evidence in either direction.
- This adjusted picture of what was met, what failed, and what remains unknown is what feeds into certainty scoring.

##### Incompleteness Disclosure

- Any evaluation concluded on incomplete data must carry an incompleteness marker that propagates through the pipeline to the frontend. The user must always know when a finding was reached without full data.
- The disclosure includes: which data points were missing, how they affected the evaluation, and what the finding would require to reach full certainty. This is not buried in metadata — it is surfaced alongside the finding in the node's dashboard.

---

### Certainty Scoring

The core of this layer. Once data completeness is resolved — either by the user providing missing data or confirming they want to proceed with what is available — the system scores how strong the evidence is behind each finding.

##### Weighted Significance

- Not all data points carry equal diagnostic weight. For some conditions, one marker is a strong primary indicator while others are supporting. The certainty model must account for *which* criteria were met, not just how many.
- A condition where the primary indicator is met but two supporting markers are missing is a stronger signal than one where three supporting markers are met but the primary indicator is absent.
- Weights must be derived from clinical literature — which data points are most diagnostically significant for each condition and evaluation. This is specific to each condition, not a global weighting system.

##### Certainty Grading

- Each finding must be assigned a certainty grade that communicates its evidential strength. The grading system must be defined once for the node and applied consistently across all evaluations.
- Grades must be meaningful and actionable — the user should immediately understand whether a finding is well-supported, partially supported, or weakly indicated due to limited data.
- The grading system must account for both the weighted significance of met criteria and the proportion of total criteria that were assessable. A finding can be high-certainty with partial data if the most significant criteria are met.

---

### How Certainty Propagates

Diagnostic certainty is not just for the user. It flows through the pipeline:

- **To Insight (Layer 4)** — the LLM must know the certainty behind each finding to contextualise appropriately. A high-certainty finding warrants different language than a low-certainty one.
- **To Pipeline (Layer 5)** — certainty grades are part of the structured output contract sent to the Orchestrator.
- **To the Orchestrator** — when performing cross-diagnostic evaluation, the Orchestrator weights node findings by their certainty. A high-certainty blood work finding carries more weight than a low-certainty one when evaluating systemic conditions.

---

### Node Implementation

When applying this framework to a specific node, document the following in that node's Evaluation section:

###### Completeness Resolution
- Define how missing data is surfaced to the user — what information is shown, what options are available, and how provided data re-enters the pipeline.

###### Incompleteness Disclosure
- Define how incomplete evaluations are marked and how that information is surfaced alongside findings in the dashboard.

###### Weighting
- For each condition and evaluation, define the diagnostic weight of each data point. Distinguish primary indicators from supporting criteria. Document the clinical rationale for weighting decisions.

###### Certainty Grades
- Define the grading system — what the levels are, what each level means, and the thresholds that determine assignment.

###### Sources
- Reference the clinical literature that supports the weighting and grading decisions.
