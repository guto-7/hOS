### Standardised Data Format

Every node must define how its data is represented internally. The standardised format is the single source of truth for the entire pipeline — Evaluation, Insight, the Pipeline layer, and the Orchestrator all read from this structure. If the format is poorly defined, every downstream layer inherits that ambiguity.

---

### What Standardisation Covers

##### Each node must define:

**Storage Schema**
- The internal data model for this node's domain. This may include database tables, file storage, or a combination — depending on the nature of the data. Tabular data (markers, measurements) fits a relational store. Large binary data (images, scans) may require file-based storage with metadata references in the database.
- Each node must define what is stored where and how the two are linked. The schema is the complete picture — not just the database, but the full storage strategy.
- Must be designed for both current use and longitudinal tracking — the same data point captured across multiple records over time should be directly comparable.
- Schema design should prioritise queryability. Evaluation and Insight both need to retrieve and compare data efficiently.

**Canonical Identifiers**
- A master list of recognised data points for this node's domain. For blood work, this is the markers list. For imaging, this is the set of recognised measurements and structures.
- Every raw data point extracted during Importing must map to an identifier on this list or be flagged as unrecognised.
- Aliases must be resolved at this stage. Different providers, labs, and manufacturers may use different names for the same data point. The node must maintain an alias mapping that resolves all known variations to a single canonical identifier.

**Unit Standardisation**
- A single standardised unit for each data point. All values are converted to this unit during Unifying, regardless of what unit the source used.
- The choice of standard unit must be documented with rationale — typically the most widely used in clinical literature for that domain.
- Conversion logic must be deterministic and well-tested. Unit conversion errors propagate silently and corrupt every downstream analysis.

---

### Design Considerations

**Extensibility**
- The schema must accommodate new data points being added without restructuring. New markers, new measurement types, new sub-mediums should be additive, not breaking.

**Provenance**
- Each stored data point should retain a reference to its source — which record it came from, when it was uploaded, what the original value and unit were before conversion. Raw values are never discarded, only supplemented with standardised equivalents.

**Cross-Node Consistency**
- While each node defines its own schema, the structural patterns should be consistent enough that the Orchestrator can consume outputs from any node without node-specific parsing logic. This is defined further in [5. Pipeline (Data Feed)](../5.%20Pipeline%20(Data%20feed)).

---

## Node Implementation

When applying this framework to a specific node, document the following in that node's Unifying section:

###### Storage Schema
- Define the data model — storage mechanisms, tables, fields, relationships, file storage strategy, and types. Include rationale for structural decisions.

###### Canonical Identifiers
- Provide the master list of recognised data points for this domain. Document the source of this list and the criteria for inclusion.

###### Alias Mapping
- Document known aliases across providers and how they resolve to canonical identifiers.

###### Unit Standards
- For each data point, define the standardised unit, the rationale for choosing it, and the conversion logic from common alternative units.

###### Provenance Fields
- Define what source metadata is retained alongside each standardised data point.
