### Validation

Validation is the integrity gate between raw extraction and trusted data. Importing rejects bad files — Validation catches bad data. A correctly formatted PDF can still produce a haemoglobin value of 500 g/L. A valid DICOM file can still contain a measurement artifact. These are not diagnostic findings — they are extraction errors that must be caught before they enter the standardised store.

---

### Why Validation Is Separate From Importing

Importing's quality thresholds operate on the source file — is it corrupt, illegible, or in a wrong format? Validation operates on the *parsed content* — does what we extracted actually make sense? This distinction matters because a file can pass every Importing check and still produce unreliable data after extraction.

Data that fails validation does not proceed to Initial Flagging or Evaluation. It is either sent to the resolution interface for user review or held until resolved.

---

### What Validation Covers

##### Each node must define checks across three categories:

**Plausibility**
- Are extracted values within physically or biologically possible ranges? Every data point in a node's domain has hard boundaries that no valid result can exceed. A negative blood glucose reading is impossible. A bone density measurement of 10,000 is an artifact.
- These are not reference ranges — reference ranges define what is clinically normal. Plausibility ranges define what is physically real. A haemoglobin of 200 g/L is implausible. A haemoglobin of 180 g/L is plausible but critically abnormal — that is for Initial Flagging to annotate, not Validation to reject.

**Completeness**
- Does the parsed record contain enough data to be useful? Each node must define the minimum required fields for a record to proceed.
- A blood work record with a test date but no marker values is incomplete. An imaging record with pixel data but no acquisition parameters may be unusable at Evaluation.
- Completeness thresholds should distinguish between hard requirements (record is rejected without these) and soft requirements (record proceeds but is marked as partial).

**Consistency**
- Do the extracted values make sense in relation to each other? Some data points have known relationships that can be cross-checked.
- For blood work: calculated fields (e.g., LDL calculated from total cholesterol, HDL, and triglycerides) can be verified against directly measured values if both are present.
- For imaging: spatial relationships between measurements can flag extraction errors (e.g., a reported organ dimension that exceeds the reported scan field of view).

---

### Handling Failed Validation

##### Each node must define:

- **Failure reporting** — what causes a data point or entire record to fail validation. Must produce a clear, specific reason that the user can act on.
- **Resolution interface** — failed data is surfaced in the node's dashboard where the user can review the original source, see why validation failed, and either correct the value and resubmit or confirm rejection. The system does not silently discard data.
- **Partial acceptance** — whether a record with some valid and some invalid data points can proceed with only the valid subset while the invalid data points are sent to the resolution interface, or whether the entire record is held until all failures are resolved.

---

## Node Implementation

When applying this framework to a specific node, document the following in that node's Unifying section:

###### Plausibility Ranges
- For each canonical data point, define the hard minimum and maximum values that constitute physically possible results.

###### Completeness Requirements
- Define the minimum required fields for a record to proceed. Distinguish between hard and soft requirements.

###### Consistency Checks
- Document known relationships between data points that can be cross-verified, and what constitutes a consistency failure.

###### Resolution Flow
- Define how failed data is surfaced to the user — what information is shown, what correction options are available, and how resubmitted data re-enters the pipeline.
- Define partial acceptance rules: can valid data points proceed while invalid ones await resolution?
