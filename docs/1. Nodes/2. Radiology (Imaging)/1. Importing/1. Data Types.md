### Data Types — Radiology Node

This document applies the framework defined in [Determining Data Types](../../../0.%20Overview/1.%20Importing%20(Data%20Collection)/1.%20Determining%20Data%20Types.md) to the Radiology (Imaging) node.

---

### Sub-Mediums

Radiology has multiple diagnostic sub-mediums — fundamentally different imaging modalities, each producing distinct types of data requiring different models and interpretation approaches:

| Modality | Data Type | Current Support | Reasoning |
|---|---|---|---|
| **X-ray (plain radiography)** | 2D grayscale projection image | **Supported (v1)** | Most common imaging modality. Digital X-rays are readily available as exportable images. Two models implemented: 18-pathology chest screening (TorchXRayVision) and bone fracture detection (SigLIP2). |
| CT (computed tomography) | 3D volumetric slices | **Rejected (v1)** | Requires volumetric rendering, slice navigation, and windowing. Fundamentally different UI (volume viewer, not 2D image). Models require 3D architectures. |
| MRI (magnetic resonance imaging) | 3D/4D multi-sequence volumes | **Rejected (v1)** | Multiple sequences (T1, T2, FLAIR, DWI) per study. Same 3D UI requirements as CT. Sequence selection adds complexity. |
| Ultrasound | 2D grayscale + Doppler | **Rejected (v1)** | Real-time modality. Static exports lose temporal information. Highly operator-dependent — image quality varies enormously. |
| Nuclear medicine (PET, SPECT) | 3D functional imaging | **Rejected (v1)** | Requires fusion with CT/MRI for anatomical context. Niche modality with limited consumer access to raw data. |
| Mammography | 2D high-resolution grayscale | **Rejected (v1)** | Specialised viewing requirements (comparison views, magnification). Regulatory sensitivity — screening mammography has strict FDA/TGA requirements for CAD tools. |

**Why X-ray only for v1:** X-rays are 2D images that map directly to the standard image processing pipeline (load → validate → standardise → infer). Every other modality requires either volumetric handling (CT, MRI, PET), specialised viewing (mammography), or temporal context (ultrasound). X-rays also have the best availability of open-source, research-grade models — TorchXRayVision is well-validated on public datasets (CheXpert, MIMIC-CXR, NIH ChestX-ray14).

**Key difference from bloodwork:** Bloodwork has a single sub-medium (lab panel reports) with uniform output format. Radiology's sub-mediums produce fundamentally different data types that cannot share a single standardisation pipeline. Each modality would need its own Stage 2 normaliser, its own model adapters, and potentially its own UI. This is why the node architecture uses pluggable model adapters — when CT support is added, it gets its own adapter without changing the X-ray pipeline.

### Supported Models

The node uses a **pluggable model adapter** architecture. Each model is a separate Python module that consumes the same standardised input (H×W float64 [0,1] grayscale array) and returns the same output shape (`[{pathology, probability, level}, ...]`).

#### Model 1: Chest X-ray Pathology Screening

| Attribute | Detail |

| **Local inference** | Yes — model runs entirely on-device |

### Severity Classification

All model outputs are classified into a 4-level severity system:

| Level | Colour | Meaning |
|---|---|---|
| **HIGH** | Red | Strong model confidence in pathology presence |
| **MODERATE** | Orange | Moderate confidence — warrants attention |
| **LOW** | Green | Low confidence — likely not significant |
| **MINIMAL** | Grey | Very low confidence — effectively negative |

**Thresholds differ per model** (clinically appropriate cutoffs are model-specific):

| Model | HIGH | MODERATE | LOW | MINIMAL |
|---|---|---|---|---|
| Chest X-ray | >= 0.7 | >= 0.4 | >= 0.2 | < 0.2 |
| Fracture (Fractured) | >= 0.8 | >= 0.5 | >= 0.3 | < 0.3 |
| Fracture (Not Fractured) | — | — | — | Always MINIMAL |

### File Formats

#### Accepted: Raster Images (PNG, JPEG, WebP, TIFF, BMP)

The accepted input formats for v1. Justification documented in [Importing.md](Importing.md) Section 1.1.

**Why raster images:** These are what patients receive from radiology portals, email, or photo exports. PIL (Pillow) provides consistent validation and metadata extraction across all formats. The raw pixel data flows directly into the standardisation pipeline.

**Format-specific considerations:**

| Format | Considerations |
|---|---|
| PNG | Preferred — lossless, exact pixel preservation. Best for reproducibility. |
| JPEG | Lossy compression introduces artifacts. Acceptable for screening but results may differ slightly from lossless source. |
| TIFF | Supports 16-bit grayscale (common in digital radiography). Highest fidelity. |
| WebP | Modern format, increasingly common. Both lossy and lossless variants. |
| BMP | Uncompressed. Rare but supported for legacy compatibility. |

#### Excluded: DICOM (.dcm)

DICOM is the clinical standard for medical imaging. Its exclusion from v1 is a significant limitation, documented with rationale in [Importing.md](Importing.md) Section 1.1. DICOM support is the highest-priority future addition for this node.

### No Equivalent to markers.json

Unlike bloodwork (which has 62 canonical markers defined in `markers.json` as the alignment contract between stages), radiology has no equivalent reference data file. This is because:

1. **Models are pre-trained** — the pathology labels are defined by the model's training data, not by a user-configurable definition file
2. **No unit conversion needed** — pixel values are standardised by the Stage 2 normaliser (grayscale [0,1] float64), not by reference-based conversion
3. **No reference ranges** — there is no "normal range" for a probability score; severity thresholds are defined per-model in the adapter code
4. **No alias resolution** — model outputs use their own labels directly, no mapping needed

The equivalent of `markers.json`'s role (ensuring consistency between stages) is fulfilled by the **standardised pixel format**: every model adapter receives the same `(H, W) float64 [0, 1]` array and returns the same `[{pathology, probability, level}]` shape. The contract is in the data shape, not in a reference file.
